### 2주차 전처리 파이프라인 재정의

**왜 매번 재정의하나요?**

Google Colab은 런타임이 종료되면 이전에 정의한 함수가 모두 사라집니다. 새 세션을 열 때마다 필요한 코드를 다시 실행해야 합니다. 이 패턴은 실제 프로젝트에서도 동일합니다. 모듈로 분리해서 `import` 하는 것이 정석이지만, 지금은 학습 단계이므로 직접 재정의합니다.

```python
# ── 2주차에서 만든 전처리 파이프라인 ─────────
불용어_목록 = [
    "것", "수", "나", "저", "제", "그", "이", "때",
    "등", "좀", "잘", "더", "한", "안", "못", "또"
]

def 전처리_파이프라인(텍스트):
    텍스트 = re.sub(r'http\\S+', '', 텍스트)               # URL 제거
    텍스트 = re.sub(r'<[^>]+>', '', 텍스트)               # HTML 태그 제거
    텍스트 = re.sub(r'[^가-힣a-zA-Z0-9 ]', '', 텍스트)    # 특수문자 제거
    텍스트 = re.sub(r'\\s+', ' ', 텍스트).strip()           # 공백 정리
    텍스트 = re.sub(r'(.)\\1{2,}', r'\\1\\1', 텍스트)        # 반복 문자 압축
    토큰들 = okt.nouns(텍스트)                             # 명사 추출
    결과 = [w for w in 토큰들 if w not in 불용어_목록 and len(w) >= 2]
    return 결과

# 동작 확인
테스트 = "ㅋㅋ 치킨 진짜 맛있어요!!! <http://test.com>"
print("파이프라인 테스트:", 전처리_파이프라인(테스트))
# 출력: ['치킨', '맛있어요']
print("✅ 2주차 파이프라인 준비 완료!")
```

## 🤔 CH 1. 왜 텍스트를 숫자로 바꿔야 할까?

![[Pasted image 20260317230323.png]]
### 1-1. 컴퓨터의 본질적 한계

컴퓨터는 겉으로 보기엔 무엇이든 다 할 것 같지만, 내부를 들여다보면 결국 **0과 1의 전기 신호**를 계산하는 기계입니다. 모든 연산은 결국 숫자 계산으로 귀결됩니다.

```
컴퓨터가 할 수 있는 것
┌──────────────────────────────┐
│  덧셈: 100 + 200 = 300       │
│  곱셈: 5 × 7 = 35            │
│  비교: 7 > 4 → True          │
│  로그: log(100) = 4.605      │
└──────────────────────────────┘

컴퓨터가 텍스트로는 할 수 없는 것
┌──────────────────────────────┐
│  "치킨 맛있어요"              │
│      ≈                       │  ← 이 계산을 할 수 없음
│  "피자 맛있어요"              │
│                               │
│  두 리뷰가 '얼마나' 비슷한가?  │  ← 모름
│  이 리뷰의 주제는 무엇인가?    │  ← 모름
│  이 리뷰는 긍정인가 부정인가? │  ← 모름
└──────────────────────────────┘
```

텍스트 상태에서는 딱 하나만 가능합니다 — **같냐 다르냐(==)**:

```python
리뷰A = "치킨이 맛있어요"
리뷰B = "피자가 맛있어요"

print(리뷰A == 리뷰B)  # False
# 컴퓨터는 "치킨"과 "피자"만 다르고 나머지는 같다는 걸 모릅니다
# 90%는 같은 문장이지만 컴퓨터에게는 그냥 "다른 것"일 뿐입니다
```

### 1-2. 숫자로 바꾸는 아이디어 — 벡터(Vector)

![image.png](attachment:499c8b4a-0913-4484-b7e2-273fd4525a24:image.png)

텍스트를 **숫자 배열(벡터)** 로 표현하면 모든 수학 연산이 가능해집니다.

**벡터란?** 방향과 크기를 가진 숫자 배열입니다. 2D 좌표처럼 생각하면 됩니다.

```
점 A = (3, 5)   ← 2차원 벡터
점 B = (7, 2)   ← 2차원 벡터

두 점 사이의 거리 = √((7-3)² + (2-5)²) = √(16+9) = 5.0  ← 계산 가능!
```

텍스트를 벡터로 표현하는 아이디어:

```
어휘 목록 = [치킨, 피자, 배달, 맛있다, 빠르다]
              [0],  [1],  [2],   [3],    [4]   ← 인덱스(차원)

리뷰A "치킨이 맛있어요" → [1, 0, 0, 1, 0]
리뷰B "피자가 맛있어요" → [0, 1, 0, 1, 0]
리뷰C "치킨 배달 빠르다" → [1, 0, 1, 0, 1]

이제 거리와 각도를 계산할 수 있습니다!
```

### 1-3. 현실에서는 수천 차원 벡터가 됩니다

실제 뉴스 기사 100만 개를 분석하면 어휘가 수십만 개에 달합니다.

```
단어 수 = 50,000개 → 각 문서는 50,000차원 벡터가 됨

기사1 = [0, 0, 3, 0, 0, 1, 0, ..., 2, 0, 0, ..., 1, 0, ...]
         ↑거의 대부분이 0        ← 희소 행렬(Sparse Matrix)
```

대부분의 값이 0인 이런 벡터를 **희소 벡터(Sparse Vector)** 라고 합니다. sklearn 같은 라이브러리는 이 0들을 효율적으로 저장하는 방법(희소 행렬)을 사용해서 메모리를 절약합니다.

### 1-4. 텍스트 수치화가 만들어낸 세상

|서비스|활용하는 텍스트 수치화 기술|우리가 느끼는 결과|
|---|---|---|
|네이버 검색|TF-IDF 기반 검색 랭킹|검색어와 가장 관련 있는 페이지 상단 노출|
|카카오 스팸 필터|BoW 기반 패턴 감지|스팸 문자 자동 차단|
|쿠팡 상품 추천|TF-IDF 유사도|"이 상품을 본 사람이 함께 본 상품"|
|ChatGPT (LLM)|Transformer 임베딩 (TF-IDF의 후계자)|자연스러운 대화|
|RAG 챗봇|TF-IDF 또는 임베딩 검색|문서에서 답변 근거 찾기|

> 💡 **이 강의의 로드맵:**
> 
> 오늘 배우는 TF-IDF는 RAG 시스템(12~16주차)의 핵심 부품입니다.
> 
> "질문과 가장 관련 있는 문서 찾기" = TF-IDF 유사도 검색

## 📦 CH 2. BoW (Bag of Words) — 단어 주머니 모델

### 2-1. BoW의 핵심 아이디어

![[Pasted image 20260317230341.png]]

**BoW(Bag of Words, 단어 주머니)** 는 1954년에 제안된 텍스트 표현 방법으로, 텍스트를 **단어가 담긴 가방**으로 봅니다.

핵심 가정: **"어떤 단어가 얼마나 자주 나왔는지"가 중요하고, "단어의 순서"는 중요하지 않다.**

**쇼핑 장바구니 비유로 이해하기:**

```
쇼핑카트 A에는:  사과 3개, 바나나 1개, 우유 2개
쇼핑카트 B에는:  사과 1개, 딸기 2개, 주스 1개

→ "사과가 몇 개 있나?"는 알 수 있음
→ "사과를 먼저 담았나, 나중에 담았나?"는 모름 (순서 무시)

텍스트 BoW도 똑같습니다.
```

```
리뷰: "치킨이 정말 맛있고 배달도 치킨만큼 빠르다"

BoW: {
    "치킨": 2,    ← 2번 등장
    "맛있고": 1,
    "배달": 1,
    "빠르다": 1
}

→ 순서 정보("치킨이 먼저 나왔는지, 배달이 먼저 나왔는지")는 완전히 사라짐
```

### 2-2. BoW에서 "순서 무시"가 왜 괜찮을까?

![[Pasted image 20260317230358.png]]

완벽하지는 않지만, 많은 상황에서 괜찮은 이유가 있습니다.

**예시 1: 감성 분석**

```
문장1: "치킨이 맛있고 배달이 빠르다"       (긍정)
문장2: "치킨이 빠르고 배달이 맛있다"       (긍정, 다른 순서)

→ BoW 결과: 동일
→ 하지만 감성(긍정/부정) 분류에는 지장 없음 ✅
```

**예시 2: 순서가 중요한 경우 (BoW의 한계)**

```
문장1: "치킨은 좋고 서비스는 나쁘다"       (치킨 긍정)
문장2: "서비스는 좋고 치킨은 나쁘다"       (치킨 부정!)

→ BoW 결과: 완전히 동일 (같은 단어, 같은 횟수)
→ 두 문장의 의미가 완전히 반대지만 BoW는 구분 못함 ❌
```

> 이런 한계 때문에 Transformer(BERT, GPT)가 등장했습니다.
> 
> 하지만 많은 실용 문제에서 BoW와 TF-IDF는 여전히 강력합니다.

### 2-3. 어휘(Vocabulary) 만들기

![[Pasted image 20260317230412.png]]

BoW를 사용하기 위해서는 먼저 **전체 문서에 등장하는 모든 단어 목록(어휘)** 을 만들어야 합니다.

```python
리뷰들 = [
    "치킨 치킨 배달 맛있다",      # 리뷰1
    "치킨 피자 배달 빠르다",      # 리뷰2
    "피자 피자 서비스 좋다",      # 리뷰3
    "배달 서비스 빠르다 좋다",    # 리뷰4
    "치킨 맛있다 서비스 좋다"     # 리뷰5
]

# 모든 리뷰의 단어를 하나로 합치기
모든_단어 = []
for 리뷰 in 리뷰들:
    모든_단어.extend(리뷰.split())

# 중복 제거(set) + 정렬(sorted)
어휘 = sorted(set(모든_단어))
print("어휘:", 어휘)
# ['맛있다', '배달', '빠르다', '서비스', '치킨', '피자', '좋다']
```

이 어휘 목록이 **벡터의 차원(축)** 이 됩니다:

```
차원 인덱스:   0      1     2       3      4     5    6
어휘 목록:  [맛있다, 배달, 빠르다, 서비스, 치킨, 피자, 좋다]

리뷰1 "치킨 치킨 배달 맛있다":
  맛있다=1, 배달=1, 빠르다=0, 서비스=0, 치킨=2, 피자=0, 좋다=0
  → 벡터: [1, 1, 0, 0, 2, 0, 0]
```

### 2-4. BoW 행렬 (Document-Term Matrix)

모든 문서를 BoW 벡터로 표현하면 **행렬(Matrix)** 이 만들어집니다.

행(row) = 문서, 열(column) = 단어

```
          맛있다  배달  빠르다  서비스  치킨  피자  좋다
리뷰1  [   1,    1,    0,      0,     2,    0,    0  ]
리뷰2  [   0,    1,    1,      0,     1,    1,    0  ]
리뷰3  [   0,    0,    0,      1,     0,    2,    1  ]
리뷰4  [   0,    1,    1,      1,     0,    0,    1  ]
리뷰5  [   1,    0,    0,      1,     1,    0,    1  ]
```

이 행렬을 **문서-단어 행렬(Document-Term Matrix, DTM)** 이라고 합니다.

### 2-5. BoW의 문제점 — 세 가지 한계

**문제 1: 순서 정보 손실 (앞서 설명)**

**문제 2: 문서 길이 편향**

```
리뷰 A (짧은 리뷰, 5단어):   "치킨 맛있다 배달 빠르다 좋다"
  → 치킨: 1

리뷰 B (긴 리뷰, 20단어):   "치킨 치킨 배달 맛있다 포장 서비스 좋다 빠르다 ..."
  → 치킨: 2

단순히 리뷰 B가 길어서 치킨 횟수가 더 많습니다.
치킨이 이 리뷰의 핵심인지는 알 수 없습니다.
→ TF로 해결 (CH 3)
```

**문제 3: 흔한 단어 과대평가**

실제 리뷰에서 전처리를 완벽히 해도 남는 단어들:

```
"음식이 정말 맛있어요. 음식 퀄리티가 최고예요."
→ 전처리 후: ["음식", "맛있어요", "음식", "퀄리티", "최고"]
→ BoW: {"음식": 2, "맛있어요": 1, "퀄리티": 1, "최고": 1}

"음식"이 2번으로 가장 높지만, "음식"은 배달 리뷰라면 당연히 모든 리뷰에 등장합니다.
정말 이 리뷰의 특징을 보여주는 단어는 "퀄리티"나 "최고"입니다.
→ IDF로 해결 (CH 4)
```

## 📐 CH 3. TF (Term Frequency) — 비율로 공평하게

### 3-1. TF란? — "얼마나 자주?"

![[Pasted image 20260317230427.png]]

**TF(Term Frequency, 단어 빈도)** 는 특정 단어가 **해당 문서 내에서** 얼마나 자주 등장하는지를 나타냅니다.

BoW의 "횟수"를 "비율"로 바꿔서 **문서 길이에 관계없이 공평하게** 비교할 수 있도록 합니다.

```
               단어가 문서에 등장한 횟수
TF(단어, 문서) = ──────────────────────────────
                  문서의 전체 단어 수

결과 범위: 0.0 ~ 1.0
  0.0 → 이 문서에 해당 단어가 없음
  1.0 → 문서 전체가 이 단어 하나로만 이루어짐 (현실에서는 불가능에 가까움)
```

**직접 계산해보기:**

```
리뷰1: "치킨 치킨 배달 맛있다"  (총 4단어)
                   ↓
TF(치킨, 리뷰1)   = 2 / 4 = 0.500  ← 전체의 절반!
TF(배달, 리뷰1)   = 1 / 4 = 0.250
TF(맛있다, 리뷰1) = 1 / 4 = 0.250
TF(피자, 리뷰1)   = 0 / 4 = 0.000  ← 등장 안 함
```

### 3-2. TF가 문서 길이 편향을 어떻게 해결하나?

실생활 예시로 생각해봅시다:

```
[문서 A] 경제 신문 기사 (100단어)
  "반도체" 5번 등장 → BoW: 5  /  TF: 5/100 = 0.050

[문서 B] 반도체 전문 저널 (500단어)
  "반도체" 15번 등장 → BoW: 15  /  TF: 15/500 = 0.030

BoW 기준: 문서 B가 "반도체" 기사처럼 보임 (횟수 15 > 5)
TF 기준:  문서 A가 더 "반도체 중심" 기사 (비율 0.050 > 0.030) ← 더 정확!

문서 B는 단지 길어서 절대 횟수가 많을 뿐,
문서 전체에서 "반도체"가 차지하는 비중은 문서 A가 더 큽니다.
```

리뷰 실습 데이터 비교:

```
짧은 리뷰 (2단어): "치킨 맛있다"
  BoW: 치킨=1  /  TF: 1/2 = 0.500

긴 리뷰 (6단어): "치킨 치킨 배달 맛있다 빠르다 좋다"
  BoW: 치킨=2  /  TF: 2/6 = 0.333

→ BoW: 긴 리뷰가 치킨 리뷰처럼 보임
→ TF:  짧은 리뷰가 더 "치킨 중심" 리뷰 ✅ (더 정확한 판단!)
```

### 3-3. TF 계산 함수 구현

![[Pasted image 20260317230440.png]]

```python
def TF(단어, 문서):
    """
    단어  : 중요도를 계산하고 싶은 단어 (문자열)
    문서  : 단어들의 리스트 ['치킨', '배달', '맛있다', ...]
    반환값: 0.0 ~ 1.0 사이의 비율
    """
    단어_횟수  = 문서.count(단어)    # 문서에서 해당 단어가 몇 번 나왔나
    전체_단어수 = len(문서)           # 문서 전체에 단어가 몇 개 있나
    return 단어_횟수 / 전체_단어수    # 비율로 반환
```

**5개 리뷰 전체 TF 행렬:**

| 리뷰        | 맛있다   | 배달    | 빠르다   | 서비스   | 치킨        | 피자        | 좋다    |
| --------- | ----- | ----- | ----- | ----- | --------- | --------- | ----- |
| 리뷰1 (4단어) | 0.250 | 0.250 | 0.000 | 0.000 | **0.500** | 0.000     | 0.000 |
| 리뷰2 (4단어) | 0.000 | 0.250 | 0.250 | 0.000 | 0.250     | 0.250     | 0.000 |
| 리뷰3 (4단어) | 0.000 | 0.000 | 0.000 | 0.250 | 0.000     | **0.500** | 0.250 |
| 리뷰4 (4단어) | 0.000 | 0.250 | 0.250 | 0.250 | 0.000     | 0.000     | 0.250 |
| 리뷰5 (4단어) | 0.250 | 0.000 | 0.000 | 0.250 | 0.250     | 0.000     | 0.250 |

리뷰1에서 치킨이 0.500, 리뷰3에서 피자가 0.500으로 각 리뷰의 핵심 단어가 잘 드러납니다.

### 3-4. TF만으로는 충분하지 않은 이유

TF가 해결한 것과 아직 남은 문제:

```
✅ 해결: 문서 길이 편향 (짧은 문서 vs 긴 문서 공평 비교)

❌ 미해결: "배달"은 5개 리뷰 중 3개에 등장
   모든 배달 리뷰에 흔히 나오는 단어 = 특정 리뷰를 구별하는 데 별 도움 없음
   TF(배달, 리뷰1) = 0.250
   TF(배달, 리뷰2) = 0.250
   TF(배달, 리뷰4) = 0.250
   → 세 리뷰에서 동일한 값 → 차별화 안 됨

해결책 → IDF (CH 4)
```

## 🔍 CH 4. IDF (Inverse Document Frequency) — 희귀함의 가치

![[Pasted image 20260317230549.png]]
### 4-1. IDF가 필요한 직관적 이유

**도서관 사서 비유로 생각하기:**

```
도서관에 책이 10,000권 있습니다.

사서 A: "사랑"이라는 단어가 들어있는 책을 찾아드릴까요?
  → 9,500권에 "사랑" 포함 (흔한 단어)
  → 이 단어만으로는 원하는 책을 찾기 어려움

사서 B: "양자 컴퓨팅"이라는 단어가 들어있는 책을 찾아드릴까요?
  → 23권만 "양자 컴퓨팅" 포함 (희귀한 단어)
  → 이 단어가 있으면 원하는 책 범위가 크게 좁혀짐!
```

**IDF의 핵심 아이디어:**

희귀한 단어일수록 **그 단어가 등장한 문서의 주제를 잘 대표**합니다.

### 4-2. IDF 공식

```
                    전체 문서 수 (N)
IDF(단어) = log( ─────────────────────────── )
                  단어가 등장한 문서 수 + 1
```

- **분자(N)**: 전체 문서 수 → 일정한 값
- **분모(df + 1)**: 단어가 등장한 문서가 많을수록 분모가 커짐 → IDF 감소
- **+1**: 분모가 0이 되는 경우 방지 (아직 아무 문서에도 없는 단어도 처리)
- **log**: 값의 범위를 압축 (아래에서 자세히 설명)

**직관적으로 표현하면:**

```
흔한 단어  →  분모 큼  →  N/df 작음  →  log(작은수) 작음  →  IDF 낮음
희귀 단어  →  분모 작음 →  N/df 큼   →  log(큰수)  큼    →  IDF 높음
```

### 4-3. 로그(log)를 왜 사용하는가?

![[Pasted image 20260317230606.png]]

이것이 가장 많이 나오는 질문 중 하나입니다. 이유는 세 가지입니다.

**이유 1: 단어 빈도는 극단적으로 불균형하다 (지프의 법칙)**

실제 텍스트에서 단어 빈도는 매우 불균형합니다. 가장 많이 쓰이는 단어 몇 개가 전체의 대부분을 차지합니다. 이를 **지프의 법칙(Zipf's Law)** 이라고 합니다.

```
영어 말뭉치 기준 (실제 통계):
  1위 단어 "the":    전체 텍스트의 7% 차지
  2위 단어 "of":     전체 텍스트의 3.5% 차지
  3위 단어 "and":    전체 텍스트의 2.9% 차지
  ...
  10위 단어:         약 1%
  100위 단어:        약 0.1%
  10000위 단어:      약 0.001%

→ 상위 몇 개 단어가 엄청나게 흔하고, 나머지 단어는 매우 드물게 등장
```

이 극단적인 차이에 로그를 적용하면 값의 범위가 압축됩니다:

```
로그 없이:
  "the": N/df = 1,000,000/1,000,000 ≈ 1.0 (IDF ≈ 1)
  "반도체": N/df = 1,000,000/100 = 10,000 (IDF = 10,000!) ← 너무 큰 값

로그 적용:
  "the": log(1) = 0.0
  "반도체": log(10,000) = 9.21 ← 합리적인 범위로 압축
```

**이유 2: 직관적으로 이해하기 쉽게**

```
로그의 성질: log(a × b) = log(a) + log(b)

100개 문서에 등장 vs 50개 문서에 등장 → 2배 차이
log(10000/100) vs log(10000/50) = 4.6 vs 5.3 → 0.7 차이 (선형에 가깝게)

10개 문서에 등장 vs 5개 문서에 등장 → 2배 차이
log(10000/10) vs log(10000/5) = 6.9 vs 7.6 → 0.7 차이 (동일하게 처리)

→ "2배 더 희귀하다"는 개념이 값 차이로 일관되게 표현됨
```

**이유 3: 실험적으로 효과가 좋다**

1970년대 이후 수많은 정보 검색 연구에서 로그를 사용했을 때 검색 성능이 더 좋다는 것이 반복 검증되었습니다.

### 4-4. IDF 값 변화 시각화

전체 문서가 10개인 경우:

```
단어가 등장한 문서 수:    1개    2개    3개    5개    8개   10개
IDF(log(10/(df+1))):     2.30   1.61   1.10   0.69   0.22   0.00

시각화:
  1개 등장:  ████████████████████████ 2.30  (매우 희귀 → 중요!)
  2개 등장:  █████████████████ 1.61
  3개 등장:  ███████████ 1.10
  5개 등장:  ███████ 0.69
  8개 등장:  ██ 0.22
 10개 등장:  (없음) 0.00  (모든 문서에 등장 → 쓸모없음!)
```

### 4-5. IDF 계산 함수 구현

```python
def IDF(단어, 전체_문서들):
    """
    단어        : 중요도를 계산할 단어
    전체_문서들 : 모든 문서(각 문서는 단어 리스트)
                  예: [['치킨', '배달'], ['피자', '서비스'], ...]
    반환값      : IDF 점수 (높을수록 희귀한 단어)
    """
    전체_문서수 = len(전체_문서들)

    # 해당 단어가 등장하는 문서의 수 계산
    # sum(1 for ...) : 조건에 맞는 개수를 셈
    등장_문서수 = sum(1 for 문서 in 전체_문서들 if 단어 in 문서)

    # IDF 계산 (+1: 분모가 0이 되는 것 방지)
    # math.log() : 자연로그(밑이 e인 로그) 계산
    return math.log(전체_문서수 / (등장_문서수 + 1))
```

**5개 리뷰 기준 IDF 계산 결과:**

```
단어      등장 문서 수    계산식              IDF 값   해석
──────────────────────────────────────────────────────────────
맛있다        2개      log(5/(2+1))=log(1.67)  0.511    비교적 희귀
배달          3개      log(5/(3+1))=log(1.25)  0.223    흔함
빠르다        2개      log(5/(2+1))=log(1.67)  0.511    비교적 희귀
서비스        3개      log(5/(3+1))=log(1.25)  0.223    흔함
치킨          3개      log(5/(3+1))=log(1.25)  0.223    흔함
피자          2개      log(5/(2+1))=log(1.67)  0.511    비교적 희귀
좋다          3개      log(5/(3+1))=log(1.25)  0.223    흔함
```

`맛있다`, `빠르다`, `피자`가 IDF가 높습니다. 이 단어들이 등장하면 어떤 리뷰인지 더 잘 구분할 수 있습니다.

### 4-6. 실제 대규모 데이터에서의 IDF 예시

배달 리뷰 100,000개를 분석한다면:

|단어|등장 문서 수|IDF|해석|
|---|---|---|---|
|"배달"|98,000개|0.02|거의 모든 리뷰에 등장 → 쓸모없음|
|"음식"|95,000개|0.05|당연한 단어 → 쓸모없음|
|"맛있다"|60,000개|0.51|흔하지만 좋은 신호|
|"바삭"|8,000개|2.53|치킨 리뷰의 특징어|
|"쫄깃"|3,000개|3.51|특정 음식(쫄면, 피자 등) 리뷰의 특징어|
|"독버섯"|2개|10.82|이 단어가 있으면 리뷰 내용이 매우 특이|

## ⭐ CH 5. TF-IDF — 두 개념의 완성

![[Pasted image 20260317230619.png]]

### 5-1. TF와 IDF를 합치는 이유

지금까지 배운 것:

```
TF: "이 문서에서 이 단어가 얼마나 자주 나왔나?"  → 문서 내 중요도
IDF: "이 단어가 전체 문서에서 얼마나 희귀한가?"  → 전체 문서에서의 독특함
```

두 정보를 결합하면:

```
TF-IDF = TF × IDF
       = (문서 내 빈도) × (전체 문서에서의 희귀도)
       = 이 문서에서 특별히 중요한 단어의 점수
```

### 5-2. TF-IDF 공식

```
TF-IDF(단어 t, 문서 d) = TF(t, d) × IDF(t)

                              단어 t의 횟수(문서 d)        N
                        =   ─────────────────────  ×  log(──────)
                              문서 d의 전체 단어 수        df(t)+1
```

### 5-3. 4가지 경우로 이해하기

TF와 IDF 조합에 따른 4가지 경우:

```
경우 1: TF 높음 + IDF 높음  →  TF-IDF 매우 높음 ⭐⭐⭐
  예: "쫄깃"이 쫄면 리뷰에서 3번 등장 + 전체 리뷰의 3%에만 등장
  의미: 이 리뷰의 핵심어! "이 리뷰는 쫄면 리뷰다"

경우 2: TF 높음 + IDF 낮음  →  TF-IDF 낮음 ⚠️
  예: "배달"이 5번 등장 + 전체 리뷰의 95%에 등장
  의미: 자주 나오지만 모든 배달 리뷰에 나오는 단어 → 쓸모없음

경우 3: TF 낮음 + IDF 높음  →  TF-IDF 중간 🔍
  예: "하얀트러플"이 1번만 등장 + 전체 리뷰의 0.01%에만 등장
  의미: 적게 나왔지만 등장 자체가 특이 → 이 식당은 고급 식당?

경우 4: TF 낮음 + IDF 낮음  →  TF-IDF 매우 낮음 ✗
  예: "음식"이 1번 등장 + 전체 리뷰의 95%에 등장
  의미: 아무 의미 없음
```

### 5-4. 손으로 계산하는 TF-IDF

리뷰들 = ["치킨 치킨 배달 맛있다", "치킨 피자 배달 빠르다", ...] (5개)

**리뷰1 "치킨 치킨 배달 맛있다" 의 TF-IDF:**

```
단어      TF 계산           TF       IDF 계산              IDF     TF-IDF
───────────────────────────────────────────────────────────────────────
맛있다   1/4 = 0.250     0.250   log(5/3) = 0.511       0.511    0.128
배달     1/4 = 0.250     0.250   log(5/4) = 0.223       0.223    0.056
빠르다   0/4 = 0.000     0.000   log(5/3) = 0.511       0.511    0.000
서비스   0/4 = 0.000     0.000   log(5/4) = 0.223       0.223    0.000
치킨     2/4 = 0.500     0.500   log(5/4) = 0.223       0.223    0.112  ← 최고!
피자     0/4 = 0.000     0.000   log(5/3) = 0.511       0.511    0.000
좋다     0/4 = 0.000     0.000   log(5/4) = 0.223       0.223    0.000
```

`치킨`이 TF-IDF 0.112로 최고점입니다. 리뷰1의 핵심어는 **치킨**입니다.

**리뷰2 "치킨 피자 배달 빠르다" 의 TF-IDF:**

```
단어      TF      IDF     TF-IDF
────────────────────────────────
맛있다   0.000   0.511    0.000
배달     0.250   0.223    0.056
빠르다   0.250   0.511    0.128  ← 최고!
서비스   0.000   0.223    0.000
치킨     0.250   0.223    0.056
피자     0.250   0.511    0.128  ← 공동 최고!
좋다     0.000   0.223    0.000
```

리뷰2의 핵심어는 **빠르다**와 **피자**입니다. 치킨 리뷰지만 배달 속도와 피자가 이 리뷰를 특징짓는 단어입니다.

### 5-5. TF-IDF 발전 과정 완전 정리

```
 문제                         해결책            결과
────────────────────────────────────────────────────────────
BoW: 단어 횟수만 셈
  └─ 긴 문서 편향 ────────→  ÷ 전체 단어 수  → TF (비율)
       TF: 비율로 계산
         └─ 흔한 단어 과대평가 → × log(N/df)    → TF-IDF (완성!)

최종: TF-IDF = 이 문서에서 자주 나왔고 + 다른 문서에서 희귀한 단어
```

### 6-1. scikit-learn이란?

**scikit-learn(sklearn)** 은 파이썬 머신러닝의 **사실상 표준(de facto standard)** 라이브러리입니다.

```
현재: 전 세계 수백만 명이 사용
특징: - 일관된 API 설계 (모든 모델이 동일한 방식으로 사용)

      - 빠른 연산 속도 (내부는 C/C++ 최적화)

```

CH 3~5에서 직접 구현한 TF-IDF를 sklearn은 단 3줄로 처리합니다:

```python
# [우리가 직접 구현: 약 40줄]
def TF(단어, 문서): ...
def IDF(단어, 전체_문서들): ...
for 문서 in 모든_문서:
    for 단어 in 어휘:
        점수 = TF(단어, 문서) * IDF(단어, 모든_문서)
        ...

# [sklearn 사용: 3줄]
vectorizer = TfidfVectorizer()
tfidf_행렬 = vectorizer.fit_transform(문서_리스트)
어휘 = vectorizer.get_feature_names_out()
```

### 6-2. fit과 transform의 의미

sklearn의 모든 모델과 변환기는 **fit → transform** 패턴을 따릅니다.

이 패턴을 이해하면 나중에 어떤 sklearn 도구를 써도 쉽게 사용할 수 있습니다.

```
fit(학습)
  → "이 데이터를 분석해서 필요한 정보를 기억해"
  → TfidfVectorizer.fit() : 어휘 목록 만들기 + 각 단어의 IDF 계산 + 기억

transform(변환)
  → "기억한 정보를 이용해서 새 데이터를 숫자로 바꿔"
  → TfidfVectorizer.transform() : TF-IDF 행렬 생성

fit_transform(학습 + 변환 동시)
  → 훈련 데이터에서는 두 작업을 한 번에
  → 더 효율적 (내부적으로 한 번만 읽음)
```

**왜 fit과 transform을 나눌까?**

4주차 머신러닝에서 핵심적으로 필요한 개념입니다:

```
1. 훈련 데이터 10,000개로 fit 실행 (어휘와 IDF 학습)
   → 어휘 목록 50,000개 완성
   → 각 단어의 IDF 값 계산 완료

2. 새로운 데이터 1개가 들어옴 (신규 리뷰)
   → transform만 실행 (이미 학습된 어휘와 IDF 사용)
   → 새 데이터에 없는 어휘는 그냥 0으로 처리
   → 새 단어가 있어도 무시 (어휘에 없으므로)

⚠️ 중요: 새 데이터로 fit을 다시 하면 안 됩니다!
   → 새 데이터만의 IDF가 반영되면 훈련 때와 다른 기준이 됨
   → "데이터 누수(Data Leakage)" 문제 발생
```

### 6-3. sklearn 입력 형식과 ' '.join()

TfidfVectorizer는 **공백으로 구분된 문자열 리스트**를 입력으로 받습니다.

```python
# 2주차 파이프라인의 출력 (단어 리스트)
전처리_결과 = ["치킨", "바삭", "맛있어요"]

# TfidfVectorizer 입력으로 변환
# ' '.join(리스트) : 리스트 원소를 공백으로 연결하여 하나의 문자열로
문자열 = ' '.join(전처리_결과)
# "치킨 바삭 맛있어요"

# 전체 문서 리스트
문서_리스트 = [
    "치킨 바삭 맛있어요",       # 리뷰1
    "피자 치즈 도우 쫄깃",      # 리뷰2
    "배달 느리다 음식 식다",    # 리뷰3
]
```

> 🤔 **왜 리스트를 문자열로 다시 바꿔야 하나요?**
> 
> sklearn의 TfidfVectorizer는 공백을 기준으로 단어를 나눕니다. 우리가 이미 단어를 리스트로 나눠놨지만, sklearn은 내부적으로 공백 분할을 다시 하기 때문에 리스트 형태를 받지 않습니다. `' '.join()`으로 합쳤다가 sklearn이 다시 나누는 약간 비효율적인 과정이지만, sklearn의 통일된 인터페이스를 위한 트레이드오프입니다.

### 6-4. sklearn TF-IDF와 우리의 구현 차이

sklearn의 TF-IDF는 우리 구현과 세부적으로 차이가 있습니다:

|항목|우리 구현|sklearn 기본값|
|---|---|---|
|IDF 공식|log(N/(df+1))|log((N+1)/(df+1)) + 1|
|정규화|없음|L2 정규화 (벡터 크기를 1로 만듦)|
|목적|이해를 위한 단순화|실용적 성능 최적화|

```
sklearn의 IDF 공식:
  IDF(t) = log((N+1)/(df(t)+1)) + 1

+1을 분자에도 더하는 이유: 모든 문서에 단어가 있을 때 IDF=0이 아닌 1이 되도록
(완전히 흔한 단어도 최소한의 IDF 값 1을 유지)

L2 정규화: 각 문서의 TF-IDF 벡터의 크기를 1로 만들어줌
→ 문서 길이 편향 추가 보정
```

이 차이로 인해 **직접 계산한 값과 sklearn 값이 다르게 나오는 것은 정상입니다.**

### 6-5. sklearn TF-IDF 전체 사용 예시

```python

    "치킨 치킨 배달 맛있다",
    "치킨 피자 배달 빠르다",
    "피자 피자 서비스 좋다",
    "배달 서비스 빠르다 좋다",

# 1. TfidfVectorizer 객체 생성
```

## 🔗 CH 8. 문서 유사도 — 코사인 유사도

### 8-1. 문서 유사도가 필요한 상황

TF-IDF 벡터가 완성되면 문서들 사이의 **유사도(similarity)** 를 계산할 수 있습니다.

**현실에서 문서 유사도가 쓰이는 곳:**

```
검색 엔진:  "맛있는 치킨 배달" 검색
  → 쿼리를 TF-IDF 벡터로 변환
  → 수백만 페이지와 유사도 계산
  → 가장 유사한 페이지 상단 노출

표절 검사:  논문 A와 논문 B
  → 두 논문을 TF-IDF 벡터로 변환
  → 유사도가 0.9 이상이면 표절 의심 경고

뉴스 클러스터링:  오늘의 뉴스 1000개
  → 모든 기사 쌍의 유사도 계산
  → 유사한 기사끼리 묶어서 같은 사건 관련 기사 그룹화

RAG 시스템:  "한국의 인구는 몇 명인가요?"
  → 질문을 벡터로 변환
  → 지식 베이스 문서들과 유사도 계산
  → 가장 관련 있는 문단을 LLM에게 전달
```

### 8-2. 유사도 측정 방법의 종류

두 벡터가 "얼마나 비슷한지" 측정하는 방법은 여러 가지가 있습니다.

**방법 1: 유클리드 거리 (Euclidean Distance)**

```
두 점 A = (1, 2), B = (4, 6) 사이의 직선 거리

거리 = √((4-1)² + (6-2)²) = √(9+16) = 5.0

→ 거리가 가까울수록 유사 (0에 가까울수록 같음)
→ 범위: 0 ~ 무한대
```

**방법 2: 코사인 유사도 (Cosine Similarity)**

```
두 벡터 A, B 사이의 각도를 이용

코사인 유사도 = cos(θ) = (A · B) / (|A| × |B|)

               두 벡터의 내적 (dot product)
             = ─────────────────────────────
                두 벡터의 크기(길이)의 곱

→ 각도가 0°(같은 방향) → cos(0°) = 1.0  (완전히 유사)
→ 각도가 90°(직각)     → cos(90°) = 0.0  (완전히 다름)
→ 범위: 0.0 ~ 1.0 (텍스트에서 음수 벡터는 없으므로)
```

### 8-3. 왜 텍스트 분석에서는 코사인 유사도를 쓸까?

**핵심 이유: 문서 길이의 영향을 제거**

```
예시:
리뷰 A (짧음): [치킨=1, 배달=1, 맛있다=0]
리뷰 B (긴 글): [치킨=3, 배달=3, 맛있다=0]  ← A의 3배 길이, 내용은 동일

유클리드 거리:
  거리 = √((3-1)² + (3-1)² + 0²) = √8 = 2.83
  → 꽤 다른 문서처럼 보임 ❌

코사인 유사도:
  A와 B가 같은 방향을 가리키므로 각도 = 0°
  cos(0°) = 1.0  (완전히 유사!) ✅

→ 길이만 다른 문서를 동일한 문서로 올바르게 판단
```

**시각적으로 이해하기:**

```
                     ↑ (배달)
                     │
                  B ↗│
                  ↗  │
               A ↗   │
               ↗     │
──────────────────────────→ (치킨)

A = [1, 1]  (치킨 1번, 배달 1번)
B = [3, 3]  (치킨 3번, 배달 3번)

두 벡터의 방향이 완전히 같음 → 코사인 유사도 = 1.0
유클리드 거리는 차이가 있음 → 거리 = 2.83 (다른 것처럼 보임)
```

### 8-4. 코사인 유사도 직접 계산

```python
def 코사인_유사도(벡터A, 벡터B):
    """
    벡터A, 벡터B: 같은 길이의 리스트
    반환값: 0.0 ~ 1.0 사이의 유사도 점수
    """
    # 내적(dot product): 같은 위치의 원소끼리 곱해서 합산
    내적 = sum(a * b for a, b in zip(벡터A, 벡터B))

    # 각 벡터의 크기(magnitude): 모든 원소의 제곱합의 제곱근
    크기A = math.sqrt(sum(a**2 for a in 벡터A))
    크기B = math.sqrt(sum(b**2 for b in 벡터B))

    # 분모가 0이 되는 경우 방지 (영벡터 처리)
    if 크기A == 0 or 크기B == 0:
        return 0.0

    return 내적 / (크기A * 크기B)

# 테스트
A = [1, 1, 0, 0]  # 치킨=1, 배달=1, 피자=0, 서비스=0
B = [3, 3, 0, 0]  # 치킨=3, 배달=3, 피자=0, 서비스=0
C = [0, 0, 1, 1]  # 치킨=0, 배달=0, 피자=1, 서비스=1

print(f"A와 B의 유사도: {코사인_유사도(A, B):.4f}")  # 1.0000 (같은 방향)
print(f"A와 C의 유사도: {코사인_유사도(A, C):.4f}")  # 0.0000 (완전히 다름)
```

### 8-7. 코사인 유사도의 한계

코사인 유사도도 완벽하지 않습니다:

```
리뷰A: "치킨이 맛있다"    → 긍정 리뷰
리뷰B: "치킨이 맛없다"    → 부정 리뷰!!

TF-IDF BoW 기반 코사인 유사도:
  두 리뷰에서 "치킨"이 공통으로 있음
  → 유사도가 꽤 높게 나올 수 있음
  → 하지만 의미는 정반대!

이유: TF-IDF는 단어의 '유무와 빈도'만 보고, '의미'는 보지 않기 때문
해결: 의미를 이해하는 임베딩(Word2Vec, BERT) → 5주차 이후에 배웁니다
```

## 📌 전체 핵심 정리

### BoW → TF → IDF → TF-IDF 발전 과정

```
단계       방법               해결한 문제                 남은 문제
──────────────────────────────────────────────────────────────────
BoW     단어 횟수            가장 단순한 수치화          문서 길이 편향
                                                         흔한 단어 과대평가
  ↓ (÷ 전체 단어 수)
TF      비율 (0.0~1.0)       문서 길이 편향 해결         흔한 단어 과대평가
  ↓ (× log(N/df))
IDF     희귀도 가중치         흔한 단어 과대평가 해결     단어 순서 무시
  ↓ (TF × IDF)                                          의미 이해 불가
TF-IDF  문서의 핵심어 점수   두 문제 모두 해결!          →  5주차 Word2Vec,
                                                            8주차 Transformer
```

### 수식 완전 정리

```
BoW(t, d)   = 단어 t가 문서 d에 등장한 횟수

               count(t, d)
TF(t, d)    = ─────────────
               len(d)

                    N
IDF(t)      = log( ───── )     (N: 전체 문서 수, df: 등장 문서 수)
                  df(t) + 1

TF-IDF(t, d) = TF(t, d) × IDF(t)

                 A · B
cos_sim(A, B) = ───────────     (A · B: 내적, |A|: 벡터 A의 크기)
                 |A| × |B|
```

### 개념 간 관계 한눈에 보기

```
단어들
  │
  ├─ 전체 문서 내 위치 무관 ──→  BoW  → TF → IDF → TF-IDF
  │    (단순 빈도 기반)
  │
  └─ 문서 내 순서 고려 ─────→  n-gram (2주차에서 언급)
                                Word2Vec, BERT (5주차 이후)

문서 유사도
  │
  ├─ 벡터 크기 고려 ────────→  유클리드 거리 (길이 편향 있음)
  │
  └─ 방향만 고려 ───────────→  코사인 유사도 (길이 편향 없음) ← 권장
```

### 전체 파이프라인 (2~3주차 통합)

```
원본 텍스트 (노이즈 포함)
          │
          ▼ [2주차] 전처리 파이프라인
          │   정제(Cleaning)     → URL, 특수문자, 이모지 제거
          │   정규화(Normalization) → 반복문자 압축, 대소문자 통일
          │   토큰화(Tokenization) → 형태소 분석, 명사 추출
          │   불용어 제거           → 의미없는 단어 제거
          ↓
     ["치킨", "배달", "맛있다"]  (단어 리스트)
          │
          │ ' '.join()
          ↓
     "치킨 배달 맛있다"  (공백 구분 문자열)
          │
          ▼ [3주차] TfidfVectorizer.fit_transform()
          │   BoW   → 단어 횟수 세기
          │   TF    → 비율로 정규화
          │   IDF   → 희귀도 가중치
          │   TF-IDF → TF × IDF
          ↓
     {"치킨": 0.45, "배달": 0.12, "맛있다": 0.67}  (TF-IDF 벡터)
          │
          ├──→ cosine_similarity() → 문서 유사도 계산
          │
          ▼ [4주차] 머신러닝 분류기 (다음 주!)
          │   나이브 베이즈, 로지스틱 회귀
          ↓
     "긍정" / "부정"
```

### 주요 sklearn 코드 치트시트

```python
from sklearn.feature_extraction.text import TfidfVectorizer
from sklearn.metrics.pairwise import cosine_similarity

# ─── TF-IDF 벡터화 ─────────────────────────────────────────
# 입력: 공백 구분 문자열 리스트
문서들 = ["치킨 배달 맛있다", "피자 서비스 좋다", ...]

vectorizer = TfidfVectorizer()
tfidf_행렬 = vectorizer.fit_transform(문서들)  # (N문서 × M어휘) 희소 행렬
어휘 = vectorizer.get_feature_names_out()      # 어휘 목록 배열
배열 = tfidf_행렬.toarray()                    # 희소 행렬 → 일반 배열

# ─── 핵심 키워드 추출 ────────────────────────────────────────
for i, 행 in enumerate(배열):
    단어_점수 = list(zip(어휘, 행))
    단어_점수.sort(key=lambda x: x[1], reverse=True)
    상위3 = [단어 for 단어, 점수 in 단어_점수 if 점수 > 0][:3]
    print(f"문서{i+1} 핵심어: {상위3}")

# ─── 코사인 유사도 ──────────────────────────────────────────
유사도_행렬 = cosine_similarity(tfidf_행렬)    # (N × N) 대칭 행렬
# 유사도_행렬[i][j] = 문서i와 문서j의 코사인 유사도 (0.0~1.0)

# ─── fit/transform 분리 사용 (4주차 패턴 미리보기) ──────────
vectorizer.fit(훈련_문서들)                    # 훈련 데이터로만 학습
훈련_행렬 = vectorizer.transform(훈련_문서들)  # 훈련 데이터 변환
테스트_행렬 = vectorizer.transform(테스트_문서들)  # 테스트 데이터 변환
```

|`AttributeError: 'list' object has no attribute 'lower'`|TfidfVectorizer에 단어 리스트를 직접 전달|`' '.join(리스트)` 로 문자열로 변환 후 전달|
|`TypeError: expected string or bytes-like object`|리스트 안에 `None` 값 포함|`[x for x in 결과들 if x is not None]` 으로 필터링|
|`ModuleNotFoundError: No module named 'sklearn'`|scikit-learn 미설치|`!pip install scikit-learn` 실행|
|`IDF 값이 0이 나온다`|모든 문서에 해당 단어가 등장|정상 동작 — 모든 문서에 있는 단어는 IDF=0|
|`sklearn 결과가 직접 계산한 값과 다르다`|sklearn의 정규화 방식 차이|정상 동작 — 개념은 같지만 세부 구현이 다름|
