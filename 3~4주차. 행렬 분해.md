# 데이터
- Feedback: 고객이 아이템에게 어떤 액션을 취했는지에 대한 정보.

| `user_id` | `movie_id` | `rating` | `rated_time` |
| --------- | ---------- | -------- | ------------ |
| 3         | 5          | 4        | 02:00        |
| ...       | ...        | ...      | ...          |
| 12329     | 5501       | 5        | 21:00        |

> 문제점: 이렇게 표 형태로 나열하면 특정 고객과 아이템 간의 전반적인 패턴이나 특징을 한 눈에 파악하기 어렵다.
# 데이터 변환과 희소성
- 데이터 피벗: 표 형태의 데이터를 2차원 행렬로 변환하자. 행을 사용자로, 열은 영화로 두고 교차점에 평점을 배치한다. (user-item matrix)
- Sparsity 문제:
	- 사용자가 세상의 모든 영화를 볼 수는 없다. 
- Matrix Factorization: 행렬을 고객에 대한 행렬, 아이템에 대한 행렬로 분해하자.
	- 즉, 고객에 대한 행렬 $C$와 아이템에 대한 행렬 $I$에 대하여 $CI = A$ 형태가 된다.
	- SVD: 너무 느림!

# Matrix Factorization
$R (m \times n) \sim P(m \times k) \times Q(k \times n)$
- $m$: 유저 수
- $n$: 아이템 수
- $k$: 잠재 차원 수(보통 10~200)

# 추천의 핵심 문제: Latent Factor
각 유저를 $k$개의 숨겨진 취향 차원으로 표현
ex) {발라드 0.9, 힙합 0.1, 인디 0.7}

## Latent Space
유저와 아이템이 가까운 벡터 공간에 존재한다.
가까운 거리 = 높은 선호도
즉, 사람 A 벡터 근처의 아이템 = 그 사람이 좋아할 만한 곡.

가깝다 = 내적값이 커진다!

# ALS(Alternating Least Squares) 논문 읽기
## Explicit vs Implicit Feedback
- Explicit: 별점, 좋아요
	- 유저가 직접 표현
	- 정확하지만 양이 많지 않음
- Implicit: 선호도
	- 행동에서 추론
	- 양이 많지만, Noisy

=> Explicit 자료가 질이 좋지만 적다. 즉, Implicit 자료를 잘 활용할 수 있어야 한다.

## ALS
ALS는 번갈아가며 최적화를 진행한다.
- P 고정 -> Q 최적화
- Q 고정 -> P 최적화
- ... (번갈아가며 반복)

장점
- 매 스텝 Loss 감소를 보장한다.
- 병렬화가 쉽다 (by Spark)
- Implicit Feedback에 적합

## Package: `implicit`
`implicit.cpu.als.AlternatingSquares`
- ALS, BPR 등 주요 구현체 제공
- GPU 가속 지원
- 주요 파라미터
	- `factors`: 잠재 차원 수 $k$.
	- `regularization`: $\lambda$ (과적합 방지용)
	- `alpha`: $\alpha$ (확신도 스케일)
	- `iterations`: 반복 횟수
